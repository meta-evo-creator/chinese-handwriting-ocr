# Lessons Learned — 手写体OCR实战经验

> 2026-05-05 · 廉洁承诺书手写日期提取项目
> 测试材料: 20页扫描件，10份手写日期，3个OCR引擎对比

## 引擎选择教训

| 引擎 | 占用 | 手写体准确率 | 结论 |
|------|:----:|:----------:|:----:|
| Tesseract 5.4 | 240MB | 10-30% | ❌ 手写体不可用，保留给印刷体 |
| EasyOCR 1.7 + torch | 663MB | 30-50% | ❌ 中等+最重，已卸载 |
| **RapidOCR 1.4** | 55MB | **60-85%** | ✅ 轻量高性能，主力引擎 |

**教训:** EasyOCR看起来"全能"但中文手写体不如RapidOCR，却带了447MB的PyTorch。测试前应先对比再全量安装。

## 日期提取三大坑

### 1. 裁剪位置偏移
- 固定 `y=0.46-0.64` 的裁剪区域在部分页面漏了日期行
- **解决:** 全页低分OCR动态定位找"年/月/日"关键字位置
- 准确率提升: 20%(固定) → 60%(动态)

### 2. 手写数字"2"的误读
- "2022"中的"2"常被OCR读成5、7、1
- 表现为: 2025年、2027年、2021年
- **当前:** 通过值域校验(年≤2026)纠正未来年份
- **根本解:** 需要专用手写数字CRNN模型

### 3. 前导数字"1"丢失
- "15日"的"1"和"5"之间有间隙→只读到"5日"
- **当前:** 后处理检查OCR文本中附近是否有"1"→拼接
- **未来:** 连接字符分析

## 流水线设计模式

```
简单固定裁剪        → 动态行定位         (+40%准确率)
单次OCR             → 多帧投票           (+15%) 
原始OCR输出          → 智能值域校验       (+20%)
独立处理             → 上下文推断         (+10%)
```

每个环节都是可插拔的，根据场景取舍效率与准确率。

## 进程管理

**重要:** RapidOCR引擎初始化后，如果Python进程被强制终止，ONNX模型文件可能被锁住，导致后续调用hang住。

**处理方式:**
```bash
# 清理僵尸进程
Get-Process python* | Where-Object {$_.CPU -gt 10} | Stop-Process -Force
```

考虑在脚本中加超时保护或进程池管理。

## 后续改进优先级

1. **高:** 手写数字CRNN微调（基于ONNX，轻量）
2. **中:** 批次自动重试机制（OCR失败自动换参数重试）
3. **低:** 多引擎交叉验证（Tesseract+RapidOCR双确认）
