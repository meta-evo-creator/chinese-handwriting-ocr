## 核心哲学：能力马赛克化，绝不重复造轮子

> MEV Sense 原则：优先复用 ClawHub/GitHub 技能 → 权重自适应 → 动态编组最佳组合

### 正例 ✅ 今天的正确决策

```
RapidOCR 文字检测器（定位日期行位置）
   ↓
RapidOCR 文字识别（读日期文本）
   ↓
智能后处理校验（值域检查+年份纠偏+前导1推断）
```

这个马赛克组合实测 **6/10**，比单一引擎的3/10翻了一倍。

### 反例 ❌ 绕远路的决策

**从零训练CRNN模型** — 写了CRNN架构、合成数据生成器、训练pipeline，500条合成数据训练15epoch，结果在真实扫描件上准确率0%。

**问题出在哪：**
1. 没先问"有没有现成的预训练模型？" → CnOCR的number-densenet系列就是现成的
2. 直接跳到"自己造" → 合成数据分布和真实手写差异太大
3. 应该先试一行命令微调 `cnocr train -m number-densenet_lite_136-gru --train-file ...`
4. 不行再考虑从零训练

## 引擎选择教训

| 引擎 | 占用 | 手写体准确率 | 结论 |
|------|:----:|:----------:|:----:|
| Tesseract 5.4 | 240MB | 10-30% | ❌ 手写体不可用，已卸载 |
| EasyOCR 1.7 + torch | 663MB | 30-50% | ❌ 中等+最重，已卸载 |
| **PaddleOCR 2.8** | ~200MB | 20-40% | ⚠️ 文档OCR主力，手写体弱 |
| **RapidOCR 1.4** | 55MB | **60-85%** | ✅ 轻量高性能，手写体首选 |

**教训:** EasyOCR看起来"全能"但中文手写体不如RapidOCR，却带了447MB的PyTorch。测试前应先对比再全量安装。

## 最佳能力组合（当前最优解）

```
Step 1 动态日期行定位
   RapidOCR全页低分(150dpi)OCR
   → 找包含"年/月/日"的文本行
   → 动态计算裁剪坐标

Step 2 高精度日期OCR
   RapidOCR 450dpi精准识别裁剪区域
   → 提取年月日原始文本

Step 3 智能值域校验
   年份: 2020≤年≤2026, 2027→2022(5→2纠偏)
   月份: 1-12, 前导0补齐
   日期: 1-31, 前导1推断(5→15)

Step 4 上下文推断
   统计同批文档共同年月日
   补充UNKDATE结果
```

实测准确率: **6/10**（比单一引擎3/10翻倍）

## 日期提取三大坑

### 1. 裁剪位置偏移
- 固定 `y=0.46-0.64` 的裁剪区域在部分页面漏了日期行
- **解决:** 全页低分OCR动态定位找"年/月/日"关键字位置
- 准确率提升: 20%(固定) → 60%(动态)

### 2. 手写数字"2"的误读
- "2022"中的"2"常被OCR读成5、7、1
- 表现为: 2025年、2027年、2021年
- **当前:** 通过值域校验(年≤2026) + 数字替换(5→2,7→2)
- **根本解:** 真实手写样本微调

### 3. 前导数字"1"丢失
- "15日"的"1"和"5"之间有间隙→只读到"5日"
- **当前:** 后处理检查OCR文本中附近是否有"1"→拼接
- **局限:** "01"和"15"仍会混淆（如01读成15，15读成01）

## 流水线设计模式

```
简单固定裁剪        → 动态行定位         (+40%准确率)
单次OCR             → 多帧投票           (+15%) 
原始OCR输出          → 智能值域校验       (+20%)
独立处理             → 上下文推断         (+10%)
```

每个环节都是可插拔的，根据场景取舍效率与准确率。

## 进程管理

**重要:** RapidOCR引擎初始化后，如果Python进程被强制终止，ONNX模型文件可能被锁住，导致后续调用hang住。

**处理方式:**
```bash
# 清理僵尸进程
Get-Process python* | Where-Object {$_.CPU -gt 10} | Stop-Process -Force
```

考虑在脚本中加超时保护或进程池管理。
